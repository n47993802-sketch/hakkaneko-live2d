阿卡貓 HakkaNeko 網站 — 檔案總說明
============================================================
這份文件是給「未來的你」看的：半年後如果忘記某個檔案是做什麼用的，
先回來看這份文件，應該就能想起來。

────────────────────────────────────────
一、分頁（*.html）— 使用者看得到的 10 個頁面
────────────────────────────────────────

index.html
    網址：/ 或 /index.html
    導覽列顯示名稱：自我介紹
    內容：首頁，個人介紹、合作繪師快速入口、頁尾社群連結／Ko-fi。

artists.html
    導覽列顯示名稱：合作繪師
    內容：合作繪師名單與介紹卡片。

rules.html
    導覽列位置：委託資訊 ▾ → 流程與規範
    內容：委託流程、注意事項、規範說明（文字為主）。

core.html
    導覽列位置：委託資訊 ▾ → V皮設計
    內容：V皮（Live2D 建模）委託方案、報價計算機、送單表單。

anim.html
    導覽列位置：委託資訊 ▾ → 動畫設計
    內容：動畫設計委託方案、報價計算機、送單表單。

template.html
    導覽列位置：委託資訊 ▾ → 聯名模板
    內容：聯名模板（30 款衣裝）挑選介面、報價計算機。

portfolio.html
    導覽列位置：創作空間 ▾ → 作品展示
    內容：作品展示（V皮/動畫作品封面、動態貼圖、動態 Logo 三個子區塊）。
    動態貼圖／動態Logo 的資料跟渲染邏輯已拆到 portfolio-config.js／
    portfolio-render.js，只有這一頁會載入這兩個檔案。

channels.html
    導覽列位置：創作空間 ▾ → 推薦頻道
    內容：推薦的 Twitch / YouTube 頻道，含直播中狀態自動偵測。
    頻道清單跟偵測邏輯已拆到 channels-config.js／channels-render.js，
    只有這一頁會載入這兩個檔案（也因此不再像以前一樣，其他 9 個
    頁面每次載入都偷偷幫你查一次推薦頻道直播狀態、卻沒地方顯示）。

fanart.html
    導覽列位置：創作空間 ▾ → 二創展示
    內容：二創圖片展示。圖片清單不是寫死的，而是即時呼叫 GitHub API
    列出指定資料夾內容，邏輯已拆到 fanart-render.js（只有這一頁會
    載入這個檔案；因為清單本身就是即時抓的，沒有另外拆 config 檔）。

live2d-demo.html
    導覽列位置：創作空間 ▾ → 角色互動（目前「隱藏」中）
    內容：角色即時互動 Demo 的頁面骨架，Live2D 模型尚未實際串接
    （common.js 裡的 LIVE2D_MODEL_URL 目前還是預留的範例網址）。
    因為功能還沒做完，所以在 nav-config.js 裡設定 enabled: false，
    不會顯示在導覽列，但直接輸入網址還是打得開，方便你自己測試進度。
    ⚠️ 這個頁面本身沒有結構性 bug，只是功能本來就還沒做完；
    等 Live2D 模型準備好、想要正式上線時，去 nav-config.js 把這一項的
    enabled 改回 true 即可，不需要動這個 html 檔案本身。

────────────────────────────────────────
二、共用邏輯檔案
────────────────────────────────────────

nav-config.js
    整個網站「唯一」需要編輯的導覽列設定檔。
    想要隱藏／顯示某個分頁，只需要把該項目的 enabled 改成 false / true，
    不用跑去 10 個 html 檔一個一個改。
    每個頁面都必須在 nav-render.js 之前載入這個檔案。

artists-config.js　【新增，解決「繪師換頭貼要到處找」的問題】
    合作繪師名單（頭貼／名字／X 帳號連結）集中在這一份檔案的
    一個陣列裡。之前這些資料是直接寫死散落在 artists.html 裡
    18 段幾乎一樣、只有頭貼網址跟名字不同的重複 HTML 中，
    繪師換頭貼時要在三百多行裡面找到對應那一段才能改。
    現在只要打開這一個檔案，搜尋繪師的名字或 @帳號，
    把 avatar 欄位換成新的圖片網址（或本機路徑）即可。

artists-render.js　【新增】
    讀取 artists-config.js 把繪師卡片畫進 artists.html 的
    #artistsGrid 容器，做法跟 nav-render.js 一樣是「同步、立刻」
    畫出來，不會有空白閃爍。額外加了頭貼載入失敗時的備用畫面
    （改顯示一個素色人形 icon，而不是瀏覽器預設的破圖）。

nav-render.js　【新增，修復「切換分頁閃兩下」問題的關鍵】
    負責「同步、立刻」把導覽列按鈕畫出來的檔案。
    原本這份網站少了這個檔案（10 個頁面都有寫 <script src="nav-render.js">，
    但實際上這個檔案並不存在，等於每次都在跟瀏覽器要一個不存在的檔案），
    所以導覽列一開始永遠是空的，要等到頁面最下面的 common.js 執行完，
    按鈕才「突然」跑出來 —— 這就是切換分頁時感覺閃一下的主因之一。
    現在補上這個檔案後，導覽列會在畫面第一次繪製前就準備好，不再空一拍。

common.js
    全站共用的主要邏輯：多國語言（i18n）、報價計算機、燈箱看圖、
    聯名模板互動、頻道直播狀態偵測、二創圖片載入、Loader 動畫、
    日夜模式、導覽列渲染備援（如果 nav-render.js 沒載入成功時的保險）等等。
    由原本的單頁 SPA 拆分而來，所以裡面還留著少量「switchTab()」這種
    舊版單頁時代的分頁切換函式；現在每個分頁都是獨立 html、
    用真正的網址跳轉，這個函式目前沒有任何按鈕在呼叫它，
    是安全但可以之後找機會清掉的舊程式碼，先保留、不動它以策安全。

common.css
    全站共用樣式：Loader 動畫、日夜模式配色、卡片／按鈕樣式、
    燈箱樣式、RWD（響應式）調整等等。

tailwind.min.css　【新增，這次「閃兩下」的真正主因就在這裡】
    預先編譯好的 Tailwind 樣式表，取代原本的
    <script src="https://cdn.tailwindcss.com/3.4.16"></script>。
    詳細原因見下方「四、這次調整內容摘要」第 1 點，這是目前找到
    信心程度最高的成因，比 nav-render.js 缺失的影響範圍更大、
    也更能解釋為什麼委託資訊那四頁特別明顯。

effects/meteor-bg.js　【從 common.js 拆分出來】
    背景的流星／星空／星雲飄落特效，畫在每個頁面都有的
    <canvas id="meteor-canvas"> 上。以前這段程式碼混在 common.js
    正中間（一支近 3000 行的檔案裡的其中一段），現在獨立成一個檔案，
    以後想調整流星數量、速度、顏色，只需要打開這一個檔案就好，
    不用再去 common.js 裡大海撈針。
    對外只暴露 window.setMeteorMode(isLight) 這一個函式，
    由 common.js 的日夜模式切換呼叫。

locales/zh-TW.js、zh-CN.js、en.js、ja.js
    四國語言字典，各自把翻譯掛到 window.I18N['語言代碼']。
    順序不影響結果，但都必須在 common.js 之前載入。

portfolio-config.js／portfolio-render.js　【新增，只有 portfolio.html 載入】
    動態貼圖／動態Logo 的圖片清單（config）與建卡片/分頁/GIF 零閃爍
    渲染邏輯（render），從 common.js 拆分出來。要新增/修改作品，
    編輯 portfolio-config.js 的陣列即可。

channels-config.js／channels-render.js　【新增，只有 channels.html 載入】
    推薦頻道清單（config）與直播狀態偵測邏輯（render），從
    common.js 拆分出來。要增減推薦頻道，編輯 channels-config.js。

fanart-render.js　【新增，只有 fanart.html 載入】
    二創圖片的 GitHub API 讀取邏輯，從 common.js 拆分出來。
    沒有另外拆 config 檔，因為清單本身就是即時抓的，不是寫死陣列。

四、這次調整內容摘要
────────────────────────────────────────

0.【追加調查・目前信心程度最高的主因】把 Tailwind 從「CDN 即時編譯」
   改成「預先編譯好的靜態檔案」（tailwind.min.css）。
   原因：原本每個頁面都是用
       <script src="https://cdn.tailwindcss.com/3.4.16"></script>
   這種寫法載入 Tailwind，這支腳本的運作方式是——瀏覽器先把「完全
   沒套用樣式」的原始 HTML 畫出來一次，然後這支腳本才開始「即時掃描」
   整個頁面用到哪些 class，現場編譯出對應的 CSS 並插入 <head>，
   全頁瞬間重新套用樣式、整個重排一次。
   這正好可以解釋你回報的現象：
     - 為什麼委託資訊裡的「流程與規範／V皮設計／動畫設計／聯名模板」
       這四頁閃得特別明顯？因為這四頁的表單、報價計算機用了大量
       獨一無二、極少重複的 Tailwind class（peer-checked、
       各種 text-[10px] 這類自訂數值），現場掃描＋編譯的時間拉長，
       使用者看到「先是無樣式的畫面、然後整頁瞬間跳成正常樣子」的
       那個瞬間就特別明顯，感覺像閃兩下。
     - 為什麼其他頁面也有「文字彈跳／短暫跳出」但比較輕微？
       因為這些頁面用到的 class 數量少、掃描編譯快很多，
       同一個機制、只是耗時短到比較不容易注意到而已。
   修法：改用 Tailwind CLI 把整個網站實際用到的 class「離線」先編譯好
   成一份固定的 tailwind.min.css（跟 common.css 一樣是單純的靜態
   樣式表，瀏覽器一開始下載就直接套用，不需要等 JS 執行、也不需要
   現場掃描），10 個頁面全部換成
       <link rel="stylesheet" href="tailwind.min.css">
   這樣從根本上不會再有「先無樣式、後套用」這個過程，因此也不會再有
   對應的畫面跳動。這一版本已用 Tailwind CLI 重新產生
   tailwind.min.css，並抽查過所有頁面裡實際用到的特殊 class
   （含 core.html／anim.html／template.html／rules.html 裡大量的
   arbitrary value，例如 text-[10px]、bg-[#059669] 等）皆有成功
   編譯進去，沒有遺漏。
   ⚠️ 之後如果你新增了全新的 Tailwind class（例如新頁面用到
   之前從沒出現過的顏色/尺寸），必須重新執行一次編譯，
   tailwind.min.css 才會包含新 class，否則新樣式不會生效。
   最簡單的方式：把 10 個 html 檔的
       <link rel="stylesheet" href="tailwind.min.css">
   暫時換回
       <script src="https://cdn.tailwindcss.com/3.4.16"></script>
   測試新樣式沒問題後，再請我（或用 Tailwind CLI）重新編譯一次靜態檔，
   換回 <link> 版本上線，避免正式站又出現本次修好的閃爍問題。

1. 補上遺失的 nav-render.js（10 個頁面都有引用，但檔案原本不存在）。
   這是「導覽列本身」會空一拍再出現按鈕的成因，跟上面第 0 點是
   兩個獨立的問題，都已修復。
2. 把日夜模式的偵測邏輯提早到 <body> 最開頭同步執行，
   避免深色模式使用者看不到，但淺色模式使用者每次切分頁都會先
   閃一下預設深色、才變回淺色的次要成因。
3. 把背景流星特效從 common.js 中拆成獨立的 effects/meteor-bg.js。
4. 清掉 common.css 裡兩份互相打架、後者永遠蓋掉前者的
   .page-content 動畫設定（死碼，從未真正生效過的那一份已移除）。
5. 建立 images/、videos/ 資料夾骨架＋對應說明文件，
   供之後搬運與轉檔使用（尚未變更任何現有的圖片網址，網站顯示不受影響）。
6. 確認 live2d-demo.html 已經透過 nav-config.js 的 enabled:false
   正確隱藏，且該頁本身沒有額外的結構性問題。

五、第二輪調整（切換頁面卡頓／作品展示較慢／繪師頭貼維護）
────────────────────────────────────────

1.【切換分頁時流星會卡一下】
   結論：這是目前多頁架構（10 個各自獨立的 html）天生的限制，
   無法在不大改架構的情況下完全消除。原因是每次點導覽列都是
   瀏覽器「整頁重新載入」，不是像早期一體式版本那樣同一份頁面
   只是切換顯示區塊——所以流星背景（effects/meteor-bg.js）也會
   跟著整個銷毀、重新初始化一次，這個重新啟動的瞬間本來就會有
   一點點卡頓，是 SPA（單頁應用）跟目前這種多頁架構的根本差異，
   除非把整個網站改寫成「用 JS 抓取內容、局部換頁不整頁重載」的
   單頁應用（工程量大很多、也會影響到 SEO／分享連結等其他層面），
   否則沒有真正能「修好」的做法。如果只是想讓卡頓感覺輕一點，
   可以考慮之後把流星數量／星星數量調低，讓初始化更快，
   但沒辦法完全消除切換瞬間那一下重新開始的感覺。

2.【作品展示（portfolio.html）明顯比其他頁面慢一步】
   找到具體原因：這頁除了畫面上本來就有的封面圖，還會用 JS
   額外建立「動態貼圖」＋「動態 Logo」兩組 GIF 縮圖（common.js
   的 gifBuildAll()）。原本的寫法會把兩組「全部」16 張圖的網址
   一次性設定好（即使畫面一次只顯示每組 4 張），等於一進頁面就
   同時對 GitHub 發出 16 張圖的下載請求，這正是它感覺「還在補圖」
   比其他頁面慢的主因。
   已修正：改成只有「目前分頁看得到的那幾張」會立刻開始下載，
   其餘的先記住網址、等使用者真的翻到那一頁時才臨時補上，
   跟其他頁面一樣「只載入看得到的量」。
   （這裡我判斷你是希望作品展示「跟其他頁面一樣快」，而不是反過來
   讓其他頁面都變慢；如果我理解反了，跟我說一聲，我再調整方向。）

3.【合作繪師頭貼常常要跟著改，有沒有更好的做法？】
   已改用跟 nav-config.js 一樣的模式：新增 artists-config.js，
   18 位繪師的頭貼／名字／連結全部集中在這一個檔案的陣列裡，
   artists.html 本身不再包含任何寫死的繪師資料。
   之後繪師換頭貼，只需要打開 artists-config.js、搜尋他的名字，
   把 avatar 網址換掉即可，不用再去三百多行的 HTML 裡面找。
   額外好處：如果頭貼網址失效（例如繪師砍帳號、換ID），
   現在會自動顯示一個素色人形圖示代替破圖，不會讓版面看起來壞掉。
   長期建議：目前頭貼都是直接連到 X（Twitter）圖床，這類連結在
   繪師「換頭貼」時網址通常還能打開、但圖片會沒有預警地變成新頭貼
   （也就是網站上顯示的圖跟著繪師本人的最新頭貼一起變動，不一定是
   你想放的那張）；如果想要「頭貼要換由我自己決定」，可以之後把
   avatar 陸續換成本機檔案路徑（例如 images/artists/紅妻.jpg），
   我這邊沒辦法直接幫你下載 X 上的圖片（無法連線到 twimg.com），
   但只要你把圖丟進 images/artists/ 資料夾，欄位路徑換一下就好，
   render 出來的結果完全一樣。

六、第三輪調整（作品展示／推薦頻道／二創展示 分開＋深層閃爍成因）
────────────────────────────────────────

1.【作品展示／推薦頻道／二創展示 三個區塊分開】
   跟合作繪師一樣的模式，這三個分頁的資料／邏輯都搬出 common.js，
   變成只有各自那一頁才會載入：
   - portfolio-config.js（動態貼圖/動態Logo 圖片清單）
     ＋ portfolio-render.js（建立卡片、分頁、GIF 零閃爍邏輯）
   - channels-config.js（Twitch/YouTube 頻道清單）
     ＋ channels-render.js（直播狀態偵測）
   - fanart-render.js（GitHub API 動態讀取二創圖片；因為清單本身
     就是即時抓的，不是寫死陣列，所以沒有另外拆 config 檔）
   好處除了「以後改資料只要開對應那個小檔案」之外，還有一個額外
   收穫：以前 common.js 是 10 個頁面共用，這些邏輯不管你在看哪一頁
   都會偷偷執行一次（例如在 artists.html 也會對 Twitch/YouTube API
   發送查詢，只是查完找不到地方顯示，白白浪費網路請求），現在只有
   真正需要的那一頁才會執行，其他頁面變得更輕量。

2.【深層挖到「切換分頁後又閃一下」的第三個成因，這次一併修正】
   在幫你分開這三頁的過程中，發現 common.js 裡有一個從「一體式
   版本」留下來、名叫 switchTab() 的函式，雖然現在不再是給導覽列
   點擊呼叫（那已經改成 nav-render.js 的真正網址跳轉），但每個
   頁面的 window.onload（頁面「所有」資源，包含圖片，都載入完成後
   才觸發，通常會比第一次畫面顯示晚一些）仍然會呼叫它一次，而它
   原本的邏輯是：把畫面內容「隱藏」再「重新顯示」，並且對
   流程與規範／V皮設計／動畫設計／聯名模板這幾頁，還會額外強制
   重播一次淡入動畫。
   也就是說，這幾頁其實會播放「兩次」淡入動畫：一次是瀏覽器一開始
   自然的 CSS 動畫（正常、很早就播完），另一次是 window.onload
   時被這段舊程式碼「無緣無故」逼著重播一次——這正是你之前回報
   「委託資訊那四頁特別容易閃兩下」的第三個、也是最直接的成因。
   已修正：拿掉這段「隱藏→重新顯示→強制重播動畫→捲動置頂」的
   邏輯（現在每個頁面本來就只有自己這一份內容、本來就是對的，
   不需要再隱藏又顯示一次）。作品展示頁比較特別，裡面的
   .port-reveal 卡片本來就是要等 JS 執行後才會從透明變成看得見，
   這部分有保留、只是改成「只有第一次真的還沒顯示過時才觸發」，
   避免以後不小心又被重複觸發。
   這個問題影響範圍比原本以為的更廣（10 個頁面理論上都會受影響，
   只是核心/動畫/模板那幾頁因為有额外的強制動畫重播，感覺特別明顯），
   這次修正後，四頁不應該再出現閃兩下的狀況，其餘頁面的「文字彈跳」
   也會一併改善。
